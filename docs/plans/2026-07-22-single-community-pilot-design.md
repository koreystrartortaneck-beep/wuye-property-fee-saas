# 单小区全量灰度设计

**日期：** 2026-07-22  
**状态：** 已确认并完成设计审查  
**目标：** 在现有物业费 SaaS 上交付一个真实小区全部业主可使用、资金链路可审计、支持退款与日对账的灰度版本。

## 1. 已确认边界

- 单个真实小区一次性向全部业主开放，不进行邀请制分批。
- 房屋绑定同时支持微信手机号精确匹配和人工申请审核。
- 账单同时支持管理人员导入和收费规则生成。
- 收费项目名称与分类可自定义，计算逻辑仅使用受控模板；特殊金额通过导入处理。
- 首版支持物业费、车位费、水电费、公摊费、滞纳金和临时收费。
- 每张新账单单独生成一笔支付订单，不合并支付。
- 首版退款只支持整笔订单全额退款；退款成功后原账单锁定为“已退款”，需要继续收费时重新开具新账单。
- 支持线下缴费录入、微信每日自动对账、电子收据和开票申请。
- 使用微信订阅消息发送账单、支付、退款和开票结果。
- 提供平台、租户及小区级暂停收款开关。
- 所有物业后台账号均可执行账单、作废和退款操作；操作一次确认后执行，不做双人复核或 MFA。
- 管理后台公网开放并使用账号密码登录；仍强制 HTTPS、强密码、登录限流、失败锁定、会话失效和审计日志。
- 不另建备份系统，依赖云数据库现有备份能力；上线前必须核查并记录实际备份与恢复能力。
- 账号注销后匿名化非必要身份信息，保留必要财务与审计记录。

## 2. 总体架构

继续使用现有技术栈：NestJS、Prisma、MySQL、Vue 3、微信原生小程序、微信云托管和普通商户微信支付 API v3。采用增量扩展，不重写现有认证、租户隔离、出账引擎和支付 Provider。

资金相关能力分为六个边界：

1. **账单工作台**：导入、规则生成、预检、草稿、发布、作废和重开。
2. **收款控制**：平台、租户及小区级开关，只阻止新订单，不阻断在途查单、回调、退款和对账。
3. **支付与线下核销**：一张新账单一笔订单，线上和线下使用同一原子核销边界。
4. **退款**：独立退款聚合、微信退款 API v3、退款通知、主动查单和幂等终态处理。
5. **对账与审计**：订单恢复、日终渠道对账、差异处理和不可变操作日志相互独立。
6. **业主服务**：手机号绑房、账单支付、收据、开票申请、退款状态和订阅通知。

当前微信商户配置是部署级全局凭据。灰度部署必须锁定允许收款的 `tenantId/communityId`，并在订单、退款和对账批次中固化 `merchantAccountId/mchid/appid`。其他租户不能借用该商户收款；未来扩展时重新决定独立商户或服务商模式。

## 3. 数据模型

### 3.1 账单状态与批次

扩展 `BillStatus`：

```text
DRAFT -> UNPAID -> PAID -> REFUNDING -> REFUNDED
              \-> CANCELED
```

- `DRAFT`：只在管理后台可见，可编辑。
- `UNPAID`：已发布且业主可见，业务字段全部锁定。
- `PAID`：线上或线下核销成功。
- `REFUNDING`：退款处理中，不允许重新支付或再次退款。
- `REFUNDED`：退款成功，原账单永久锁定。
- `CANCELED`：未支付账单作废，永久锁定。

新增统一 `BillBatch`，来源仅为 `RULE` 或 `IMPORT`。批次保存租户、小区、来源、规则或导入文件信息、文件哈希、总行数、有效行数、异常行数、总金额、状态、创建人和发布时间。`RULE` 必须关联规则；`IMPORT` 必须关联文件摘要，两类字段在服务层和数据库约束中互斥。

`Bill` 增加：

- `batchId`：新批次外键。
- `source`：`RULE`、`IMPORT`。
- `sourceRowKey`：导入幂等键。
- `publishedAt/publishedBy`。
- `canceledAt/canceledBy/cancelReason`。
- `replacesBillId`：重开账单指向原账单。
- `ruleId` 改为可空，导入账单不伪造规则。
- `billRunId` 暂时保留为可空兼容字段，待旧数据完成迁移后再移除。

迁移先为旧账单回填 `source=RULE`、`publishedAt=createdAt` 和对应 `BillBatch`，再要求新写入使用 `batchId`。现有 `(ruleId, houseId, period)` 唯一约束改为来源相关幂等：规则批次使用 `(batchId, houseId)`，导入使用 `(batchId, sourceRowKey)`。重开账单拥有新 ID 并通过 `replacesBillId` 关联。

发布后冻结房屋、费用项目、分类、账期、到期日、金额、来源和快照，只允许状态及审计字段变化。

### 3.2 收费项目与受控模板

`FeeRule` 增加费用分类，并继续承载名称和计算模板。允许模板：固定金额、按面积、按用量、按比例分摊。滞纳金可关闭，也可使用固定金额或受控规则生成草稿；发布后不随日期动态改变。

禁止新建或修改任意表达式 `FORMULA`。旧公式规则必须迁移为受控模板，不能迁移的改走账单导入。导入账单在快照中固化费用名称、分类和金额依据。

### 3.3 收款策略

新增分层策略：

- 平台紧急开关：仅平台超管操作，覆盖所有租户。
- 租户开关：该物业全部后台账号可操作。
- 小区开关：该物业全部后台账号可操作。

策略保存状态、变更人、原因、变更时间和可选恢复时间。有效策略要求三层全部开放。支付创建事务必须锁定并重读策略，再预占账单；以预占事务提交时间区分新订单和在途订单。

暂停只拒绝新微信支付订单。在途回调、主动查单、关单、退款和对账继续运行。线下补录只允许登记停收前已实际发生的付款，必须填写实际付款时间、凭证号和原因。

### 3.4 支付与历史兼容

`Payment` 增加：

- `tenantId/communityId/billId`，新订单必须关联一张账单。
- `merchantAccountId/mchid/appid` 快照。
- `transactionId` 唯一索引。
- `confirmedBy`：`WXPAY_NOTIFY`、`WXPAY_QUERY`、`OFFLINE`、`MOCK`。
- `wxpayNotifiedAt`：有效微信支付通知到达时间，即使订单已由主动查单先行确认也要记录。
- `expiresAt/closedAt/lastSyncedAt`。
- `failureCode/failureMessage`。
- `receiptNo/receiptSnapshot`。
- 线下凭证号、实际付款时间、操作人、付款人快照和备注。
- `wxUserId` 改为可空，仅线下渠道允许为空。

现有 `PaymentBill` 暂时保留，只读兼容历史多账单订单；新订单同时写 `Payment.billId`，不再创建多张关联。一个账单可以有多个历史支付尝试，但仅可通过现有 `Bill.paymentId` 保留一个活动订单，不能对 `Payment.billId` 建终身唯一约束。

历史合并订单若全额退款，所有关联账单同时进入 `REFUNDED`。新接口只接受单个 `billId + requestId`。

扩展支付状态表达预下单不确定结果。网络超时等不确定结果不能立即释放账单，订单进入 `PREPAY_UNKNOWN` 并主动查单；只有微信明确不存在、关闭或失败后才释放。

### 3.5 线下核销

扩展 `PaymentChannel` 增加 `OFFLINE`。线下核销事务必须原子匹配：

```text
Bill.status = UNPAID AND Bill.paymentId IS NULL
```

若存在活动微信订单，必须先主动查单；确认未支付并在微信关单后才能线下核销。线下核销创建 `SUCCESS` 支付并生成收据快照。线下冲正也走全额退款聚合，成功后支付和账单锁定为 `REFUNDED`，不恢复 `UNPAID`。

### 3.6 退款聚合与重试

新增 `Refund`：租户、小区、商户快照、稳定商户退款单号、原支付订单号、微信退款单号、原订单金额、退款金额、币种、原因、申请人、各阶段时间、通知时间、最后查询时间和失败信息。

退款状态：`CREATED`、`PROCESSING`、`SUCCESS`、`FAILED`、`CLOSED`、`ABNORMAL`。首版每个成功支付订单只有一个全额退款聚合，金额从原订单服务端派生，不能由客户端输入。

新增 `RefundAttempt` 或持久化出站任务记录每次调用。进程在调用微信前退出时使用同一商户退款单号恢复；终态失败后才允许受控重试，不能创建第二个并行退款聚合。

退款发起事务先把账单改为 `REFUNDING`。微信明确拒绝且不会继续处理时恢复 `PAID`；微信退款成功通知或主动查询确认后，原支付订单改为 `REFUNDED`、账单改为 `REFUNDED`。不能自动恢复待缴。

### 3.7 幂等、对账与审计

支付、退款、线下核销和发布使用 `(actor, action, requestId)` 唯一幂等记录，并保存请求体哈希。同键同载荷返回原结果，同键不同载荷拒绝。

现有每 10 分钟的 CREATED 订单扫描更名为“支付恢复”，不再称为财务对账。

新增：

- `ReconciliationRun`：租户、小区、商户、上海时区账单日、账单类型、渠道文件哈希、渠道与本地汇总、状态和时间。
- `ReconciliationItem`：租户、小区、业务订单号、微信交易号、差异类型、本地值、渠道值、处理状态、处理人和备注。
- 差异类型：渠道缺失、本地缺失、金额不一致、状态不一致、退款不一致。

微信适配器下载交易账单和退款账单。跑批唯一键为 `(merchantAccountId, Asia/Shanghai账单日, billType)`，支持账单延迟重试和多实例任务租约。差异项按业务单号和类型唯一，可进入自动补确认、人工关闭或升级处理。

新增 append-only `AuditLog`：租户、小区、操作者、动作、对象类型与 ID、原因、请求 ID、IP、User-Agent、变更前后摘要和时间。MySQL 触发器或独立最小权限账号拒绝应用层更新与删除，不能只依赖“不提供接口”。

新增 `PaymentEvent` 保存支付和退款有效通知摘要、确认来源及幂等结果，不保存私钥、APIv3 密钥、完整手机号、JWT 或完整敏感原文。

所有新租户域表都显式带 `tenantId`，需要时带 `communityId` 和索引，并加入 Prisma 租户扩展白名单。后台使用 `prisma.t`；回调与定时任务使用 `raw` 时必须由业务订单和商户快照显式限定。

## 4. 账单流程

### 4.1 导入

后台支持 `.xlsx` 和 `.csv`。服务端使用结构化解析器，不用字符串拆分 CSV。导入分两步：

1. 上传并预检，返回总户数、账单数、总金额和逐行异常。
2. 管理员确认后生成草稿，不自动向业主展示或发送通知。

导入列至少包含小区、房屋编码、费用项目、账期、金额、到期日和外部流水键。管理员可以在草稿阶段修正。异常批次不允许部分自动发布。

### 4.2 规则生成

规则和定时任务只生成 `DRAFT`，不立即通知。所有业主账单列表、详情、筛选、汇总和收据都要求账单已发布；直接访问草稿 ID 返回 404。管理看板排除草稿，并单列退款金额和净实收。

发布事务原子更新批次与账单，同时写入通知 Outbox；后台异步发送，避免数据库提交后进程退出导致漏通知。

修复现有规则编辑界面与后端契约不一致的问题。

### 4.3 发布、作废与重开

发布前校验户数、笔数、总金额、重复记录、房屋缺失、异常金额和历史已缴冲突。发布后账单不可覆盖编辑。

未支付账单可填写原因后作废。需要继续收费时创建新账单，并关联被作废或已退款账单；禁止覆盖原记录。

## 5. 支付与退款流程

### 5.1 支付

1. 小程序提交 `billId + requestId`。
2. 后端校验绑定、已发布账单状态、商户允许范围和三层收款开关。
3. 数据库事务锁定收款策略，原子预占账单并创建本地订单。
4. 调用微信 JSAPI 预下单并返回支付参数。
5. 微信通知和小程序主动查单竞争时使用同一幂等入账事务。
6. 支付确认事务生成唯一收据号和不可变收据快照。
7. 只有后端确认微信 `SUCCESS` 后，客户端显示成功和收据。

### 5.2 退款

1. 管理员从成功订单发起全额退款，填写原因并一次确认。
2. 后端使用稳定退款号创建退款聚合和出站尝试，再调用 `/v3/refund/domestic/refunds`。
3. 微信退款通知使用独立回调入口，完成验签、时间窗校验和 AES-GCM 解密。
4. 回调或主动查询确认 `SUCCESS` 后原子更新退款、支付和账单。
5. 退款后收据保留但显示“已作废”，原账单显示“已退款”。
6. 未开具的开票申请自动取消；已经开具的发票生成红冲待办，完成后记录红冲结果。
7. 需要重新收费时由管理员创建替代账单。

## 6. 开票、收据与通知

新增 `InvoiceApplication`，显式带租户、小区并关联成功支付订单，状态为 `SUBMITTED`、`PROCESSING`、`ISSUED`、`REJECTED`、`CANCELED`、`REVERSAL_REQUIRED`、`REVERSED`。业主提交抬头类型、抬头、税号和接收方式；物业在后台记录处理结果。系统不直接开具税务发票。

收据查询只读取支付确认时固化的 `receiptSnapshot`，包括收据号、收款方、房屋、费用项目、账期、金额和支付时间，并明确“非税务发票”。退款只附加作废状态，不能重写历史快照。

通知使用 Outbox 和幂等发送记录，覆盖账单发布、临近到期、绑定审核、支付成功、退款成功和开票完成。没有微信模板 ID或用户未授权时，业务事务仍成功，只记录通知未发送原因。

## 7. 身份、后台与小程序

手机号绑定保存独立验证证据，包括手机号哈希/脱敏值、验证时间和来源。人工审核绑定保留 `manualApprovedAt/manualApprovedBy`，手机号更新时只撤销“唯一证据为旧手机号匹配”的绑定，不能覆盖或撤销人工审核绑定。

管理员增加 `failedLoginCount/lockedUntil/tokenVersion/passwordChangedAt`。登录按 IP 和账号限流；资金请求回查账号启用状态、当前租户和 tokenVersion，停用账号或已撤销会话不能继续使用旧 JWT。所有物业账号仍拥有已确认的资金操作权限。

管理后台新增或重构：账单工作台、收款设置、支付与线下流水、全额退款、对账中心、开票申请和审计日志。

小程序调整：

- 移除账单多选和“全部缴纳”，每张账单独立支付。
- 首页多笔待缴时进入账单列表。
- 收款暂停时展示状态并禁止新下单。
- 支付记录显示退款和开票状态。
- 成功页及收据只接受后端 `SUCCESS`。
- 增加开票申请与记录页面。
- 真机验证已审核通过的手机号授权链路。

## 8. 安全、备份与故障处理

- 管理后台生产入口必须 HTTPS-only；HTTP 禁用或跳转是发布硬门槛。
- 所有资金接口使用请求幂等键，金额以整数分参与渠道比较。
- 支付和退款通知失败返回微信要求的真实 HTTP 状态；成功返回 HTTP 200。
- 全局停收发生时，在途支付仍可入账，避免“用户已付款但系统拒收”。
- 告警覆盖回调拒绝、退款异常、对账差异、长期待确认订单和任务失败。
- 不新增独立备份系统，但迁移前必须确认实际数据库为云托管 MySQL，并记录自动备份是否启用、保留周期、最近成功时间、可恢复时间点、恢复权限和预计 RPO/RTO；至少形成一次云厂商恢复流程验证记录。

- 账单状态机、导入幂等、发布原子性、草稿不可见和替代账单关联。
- 单账单单订单、请求幂等、并发下单和停收竞态。
- 支付通知与主动查单乱序、重复通知和不确定预下单。
- 全额退款发起、进程中断恢复、失败重试、通知解密、主动查询和乱序幂等。
- 线下核销与活动微信订单竞争。
- 日对账渠道取数、五类差异、重复跑批和任务租约。
- 开票与退款联动、不可变收据快照。
- 注销匿名化、旧手机号自动绑定撤销和跨小区访问隔离。
- 管理员锁定、会话撤销、登录限流和审计不可修改。
- 管理后台关键页面构建和小程序静态流程测试。

生产验收至少包括：真实手机号授权、0.01 元单账单支付、真实全额退款、退款通知审计、收款暂停、线下核销、日对账和开票申请。测试数据使用独立联调小区，不修改正式业主账单。

## 10. 发布与成功标准
