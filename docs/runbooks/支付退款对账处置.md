# 支付 · 退款 · 对账 处置 Runbook（24 小时事件流程）

> 真实收款的日常运维与异常处置。金额一律整数分；所有金融变更在幂等事务内并留审计。

## 1. 支付确认的三条路径
1. **微信支付回调**（notify）：验签 + AES-GCM 解密后入账，写 `PaymentEvent` 证据与 `wxpayNotifiedAt`。
2. **主动查单**：小程序支付完成后后端向微信查单确认。
3. **定时兜底**：每 10 分钟处理超过 30 分钟的 `CREATED` 订单（查单→确认/关单）。
> 任一路径成功即幂等入账；`PREPAY_UNKNOWN` 交由查单/兜底收敛。

## 2. 订单状态与含义
`CREATED`（下单未支付）→ `SUCCESS`（已确认）/`CLOSED`（超时关单）/`PREPAY_UNKNOWN`（预支付结果未知，待查单）/`REFUNDED`（已全额退款）。
账单：`UNPAID`→`PAID`→（退款）`REFUNDING`→`REFUNDED`；草稿 `DRAFT`；作废 `CANCELED`。

## 3. 退款处置（仅全额）
- 入口：后台「支付/退款」工作台，须填原因，幂等键防重。
- 流程：发起 → `RefundService` 调微信退款/查退款 → 退款回调或查退款确认 → 锁 `Payment.REFUNDED` + 账单 `REFUNDED`（同一幂等事务）。
- 网络不确定：保留 `PROCESSING`，由退款恢复任务（`lastQueriedAt` 乐观锁认领）查退款收敛，**不重复发起**。
- 明确 4xx 拒绝：不回滚已 `PAID` 账单、订单保持 `SUCCESS`，转人工核查。
- 联动：未开票→发票申请 `CANCELED`；已开票→ `REVERSAL_REQUIRED` 生成红冲任务。

## 4. 每日对账
- 定时下载微信对账单，与本地交易/退款比对，差异落 `ReconciliationItem`：
  `CHANNEL_MISSING`（本地有微信无）/`LOCAL_MISSING`（微信有本地无）/`AMOUNT_MISMATCH`/`STATUS_MISMATCH`/`REFUND_MISMATCH`。
- 差异项状态：`OPEN`→`AUTO_RESOLVED`/`MANUALLY_CLOSED`/`ESCALATED`。后台「对账」工作台逐项处置并留审计。
- **每日必须清零 OPEN 差异**（或升级为事件）。

## 5. 告警与事件（24h 流程）
- 告警源：回调验签失败、退款恢复耗尽、对账差异、定时任务失败、outbox 积压。去重后经 HTTPS webhook 投递（脱敏）。
- 事件：`OPEN`→`ACKNOWLEDGED`→`RESOLVED`（幂等，复发重开），全程审计。
- 指标看板（近 30 日）：技术成功率、重复扣款数、未处置对账差异、退款完成率、严重事件数、资损额 + 按日明细 + 阈值。
- **响应时限**：资损/重复扣款类告警 24h 内确认并处置；对账差异当日处置。

## 6. 常见异常速查
| 现象 | 处置 |
|---|---|
| 回调一直验签失败 | 核对 APIv3 密钥/平台证书序列号；探针 `/payment/wxpay/notify` 应 401；查商户平台证书轮换 |
| 订单卡 CREATED/PREPAY_UNKNOWN | 等兜底查单（≤10min）；仍不决则后台手动查单 |
| 业主报已付款但账单未更新 | 后台按订单号查单；确认回调是否到达（容器最小实例=1 保证在线） |
| 退款卡 PROCESSING | 等退款恢复任务；查微信退款单状态；勿重复发起 |
| 对账差异 | 按差异类型逐项核对，处置留审计，必要时升级事件 |
