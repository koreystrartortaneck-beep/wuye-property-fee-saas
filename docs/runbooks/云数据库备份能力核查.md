# 云数据库备份能力核查 Runbook

> 生产迁移/部署**前置硬门禁**：以下证据缺任意一项，即中止生产迁移与上线（本项目不自建独立备份系统，依赖云托管 MySQL 自带能力，但必须核实其真实可用）。

## 核查对象
- 实例：云托管内置 MySQL，库 `property_fee`（内网 `10.9.104.90:3306`）。

## 必须逐项确认并留证（截图/记录，不含密钥）
1. **实例标识**：控制台中的 MySQL 实例 ID / 规格。
2. **最近一次成功备份**：时间戳、状态=成功、备份大小非空。
3. **备份保留周期**：自动备份保留天数（记录具体值）。
4. **可恢复时间点**：最早可恢复点与最新可恢复点（PITR 范围，若支持）。
5. **恢复权限**：当前账号具备发起「按时间点/按备份集恢复」的权限。
6. **RPO/RTO 预期**：预计数据丢失窗口（RPO）与恢复耗时（RTO），记录数值。
7. **恢复演练证据**：至少一次「恢复到临时实例」验证可行（不覆盖生产），记录结果。

## 判定
- 全部 ✅ → 允许进入《单小区灰度发布与回滚》的三阶段发布。
- 任一 ❌ / 无法取证 → **停止生产迁移与部署**，先补齐备份能力，再上线。

## 备注
- 迁移均为加法、可独立部署，正常路径无需恢复；备份是**极端故障兜底**。
- 恢复后需重新核对：审计触发器（AuditLog 禁 UPDATE/DELETE）是否随恢复重建；金额/行数高水位是否一致。
