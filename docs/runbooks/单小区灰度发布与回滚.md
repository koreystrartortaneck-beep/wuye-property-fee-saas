## 1. 迁移清单（全部增量、向后兼容、已应用即不可改）
按序：`finance_expand` → `audit_guards` → `admin_session_hardening` → `owner_identity_hardening` → `operational_alerts`。
以上均为「加列/加表/加索引/加触发器」，不删旧列、不改历史、兼容历史多账单与跨小区支付。
`finance_switch`（收口切换：历史回填后的非空/唯一约束）**单独生成、单独部署**，不在特性分支内（见 Step 8.3）。

## 2. 发布前置（缺一不可，缺则中止上线）
1. 备份核查：见《云数据库备份能力核查》，确认最近成功备份、可恢复点、RPO/RTO、恢复权限。
2. 全量验证：`pnpm test` + `pnpm --filter @pf/api test:e2e -- --runInBand` + 小程序 `node --test` + `pnpm build` 全绿。
3. 密钥扫描：git 跟踪内容无私钥/APIv3密钥/JWT/库密码/原始回调/完整手机号。
4. 代码评审：整分支 P0/P1 全部关闭。
5. 生产配置就位：商户范围、退款/支付回调 URL、告警 webhook、订阅模板 ID、**强 JWT_SECRET**、收缴策略。
6. 后台生产 URL 必须 HTTPS。

## 3. 三阶段生产发布（维护窗口内）
**Checkpoint 1 — 加法 schema 先行**
- 仅 fast-forward `main` 到「Task 1 财务扩展」提交并推送。
- 验证旧应用在加法 schema 下继续正常服务（加法兼容）。

**Checkpoint 2 — 切到新代码路径**
- 进入维护窗口，用平台流量围栏挡住新业务流量（旧实例无法绕过）。
- 先对账并关闭全部 `CREATED/PREPAY_UNKNOWN` 订单；停定时出账；允许失败的微信通知稍后重试；把旧版本实例排空/缩容至 0。
- 跑一次性、可重启的回填任务；用行数与高水位查询确认无旧版本迟到写入。
- 部署已评审的特性分支 tip；验证新写入正常。
- 重新放量，**收缴仍保持暂停**。

**Checkpoint 3 — 收口切换迁移**
- 全部实例走新路径后，用生产预检查询生成不可变的 `finance_switch` schema + 迁移提交。
- 在最新生产数据副本上应用该提交，跑迁移专项测试 + 全量测试，取得专门 P0/P1 评审。
- 单独部署该迁移。

每个 checkpoint 均确认 CloudBase 迁移与服务健康：`/api/v1/health` 返 200、非法回调探针返真实 401。

## 4. 回滚（前向修复优先）
- 加法迁移**无需回滚**（旧应用兼容）。若新代码异常：把云托管**回退到上一在线版本**即可（schema 仍兼容）。
- 数据问题：用《备份能力核查》记录的可恢复点按平台恢复流程处理；严禁手删审计（触发器已禁 UPDATE/DELETE）。
- `finance_switch` 之后若需回退：只能前向修复（新增补偿迁移），不可逆改已应用迁移。

## 5. 发布后
- 收缴策略由 PAUSED 切 OPEN 前，先完成《单小区验收》全流程。
- 记录：Task 1 扩展提交哈希、最终评审提交哈希、各 checkpoint 时间与健康证据。
