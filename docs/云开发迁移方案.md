# 迁移到微信云开发（云函数 + 云数据库 MySQL）方案

> 目标：小程序改用 `wx.cloud.callFunction()` 走微信内部通道——**免备案、免域名、可发正式版**。
> 前提：企业主体 ✅　AppID `wx81d123b23ebbe2a4`　云环境 `cloudbase-d8gkdvv8i8d2ae64d`

---

## 目标架构

```
业主小程序 ──wx.cloud.callFunction()──→ ┐
              (免备案内部通道)           ├─→ 云函数（总路由 + 各业务域）──→ 云数据库 MySQL
管理后台Web ──@cloudbase/js-sdk────────→ ┘
```

- **一套云函数后端**同时服务业主端（小程序 SDK）与管理端（Web SDK）
- 数据库从"自有服务器 MySQL"迁到"云开发 MySQL"，彻底不依赖自有服务器

## 能保留 / 要重写

| 能直接搬（纯逻辑） | 要重写（框架相关） |
|---|---|
| 计费引擎（5 类规则、公摊算法、金额分） | NestJS controllers/guards/modules → 云函数路由 |
| 各业务状态机（工单/预约/券） | 认证：业主端改用 openid（云函数自动注入，更简单） |
| 数据模型（22 张关系表，云开发 MySQL 支持） | 多租户隔离：Prisma 中间件 → 数据访问层手动落实 |
| 错误码/枚举（shared 包） | 管理后台调用方式：HTTP → @cloudbase/js-sdk |

## 分阶段实施

### P0 · 链路验证（进行中）
- ✅ 开启 `cloud:true`、`wx.cloud.init(env)`、建 `cloudfunctions/hello`
- ⏳ **部署 hello 云函数，小程序调用成功** → 证明免备案通道通（你操作：见下方"下一步"）

### P0.5 · 关键验证：云开发 MySQL 怎么访问【最大不确定点】
先花半天确认云开发 MySQL 的访问方式，再定数据层方案：
- **若给标准 MySQL 连接串** → Prisma 直接可用，数据层几乎不改 ✅ 最理想
- **若只能走云开发 models SDK** → 需把数据访问改成 SDK 调用（工作量增加）
> 这一步结论决定后面 P2/P3 怎么写，必须先探明。

### P1 · 云开发 MySQL 建库
- 在云环境创建 MySQL 数据库
- 按现有 22 张表 schema 建表（Prisma migrate 或 SQL 脚本）

### P2 · 数据迁移
- 现有服务器 MySQL `mysqldump` → 导入云开发 MySQL
- seed 演示数据在云库重建

### P3 · 云函数后端
- 一个 `api` 总路由云函数：`callFunction('api', {module, action, data})` 内部分发
- 各业务域 handler 复用纯逻辑（计费/工单/账单/公告/服务/券…）
- 业主认证：openid 自动注入，去掉 wx-login/JWT
- 管理认证：账号密码 → 云函数校验 + 自定义登录票据
- 多租户：数据访问层统一注入 tenantId

### P4 · 业主小程序改造
- `utils/request.js`：`wx.request(http)` → `wx.cloud.callFunction()`
- 各页面调用签名不变（封装层吸收差异），页面代码基本不动

### P5 · 管理后台改造
- 引入 `@cloudbase/js-sdk`，登录云环境（自定义登录）
- `api.ts` 的 fetch → callFunction，调同一批云函数

### P6 · 联调 + 发正式版
- 全流程回归；上传 → 提交审核（物业类目按需补资质）→ 发布正式版
